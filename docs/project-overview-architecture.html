📡 Lark Agent Bridge 架构总览

面向当前 tools/lark-agent-bridge 实现的中文整体总结。重点覆盖消息入口、权限模型、路由层、执行层、状态层和 HTML 发布链路。
🎯 这是一个“飞书消息入口 + 本地分析执行 + HTML 报告发布 + reply 续聊复用”的受控桥接层。它不是通用命令机器人，重分析主要靠本地脚本，Agent 负责路由、总结和续聊复用。

A. 设计重点

BridgeApp 是唯一总调度入口
统一处理 mention、权限、续聊、资源补抓、执行、发布和回包。
日志分析与普通聊天已明显分层
signal / bug / direct / perception 走本地分析器，普通聊天走 OMLX。
intent_analysis 是可选前置层
开启后先由本地 agent 判断 route；关闭时按 parser 的确定性规则直接路由。
外部群已支持受限日志分析
非授权群的普通成员不能随便聊天，但可以用 bug 链接、reply 文件、signal、perception 这类日志分析入口。
仍有一个与本次架构无关的本地测试问题
report_server 的端口绑定测试在当前环境下会报 PermissionError。

🧠 端到端主链路

从飞书消息进入，到本地执行、HTML 发布和上下文沉淀的完整主流程。

1
消息进入
证据：lark-cli event consume im.message.receive_v1 -> LarkEvent
下游影响：进入 BridgeApp，先记活动流，再做 mention 与权限判断。
↓
2
入口判定
证据：群聊必须显式 @bot，或 reply 到已有分析消息；私聊直接进入
下游影响：决定是普通新请求、reply 续聊，还是 not_addressed 跳过。
↓
3
权限与资源补抓
证据：policy + reply_to/root_id 文件资源补抓 + 外部群日志分析例外
下游影响：把能不能进和有没有日志资源补齐，再决定具体 route。
↓
4
执行层
证据：signal / bug / direct / perception / skill / chat / followup
下游影响：本地脚本负责重分析，agent 负责路由、总结和续聊。
↓
5
发布与回包
证据：HtmlReportPublisher + LarkClient.reply/send_file_response
下游影响：统一发布 HTML 链接，必要时附带 HTML 文件。
↓
6
上下文沉淀
证据：ConversationContextStore + AgentActivityStore
下游影响：支撑 reply 续聊、复分析复用和 /sessions 控制台展示。

E. 权限框架设计图

层级放行条件能力范围
p2p默认允许全能力
allowed_users超级权限用户跨群 bypass；任何群里 @bot 都能进完整能力
allowed_chats已授权群完整能力
外部群普通成员有限放行仅日志分析类：bug / direct / signal / perception / 已有分析 followup

F. 场景框架设计图

私聊 P2P

  • 默认允许
  • 支持聊天、日志分析、skill、续聊
  • 续聊必须 reply 某条历史分析消息

已授权群

  • chat_id 在 allowed_chats
  • 完整能力开放
  • 群聊续聊必须 reply + @bot

未授权群普通成员

  • 只开放日志分析类
  • bug 链接 / reply 文件 / signal / perception / 已有分析 followup
  • 不开放通用聊天

超级用户

  • sender_id 在 allowed_users
  • 跨群 bypass allowlist
  • 任何群里 @bot 都可进入完整能力

H1. 时序图：已授权群 bug 链接分析

1
用户 → 飞书
在已授权群内 @bot 并发送 bug 链接与问题描述。
2
LarkClient → BridgeApp
消费事件，完成 @bot 识别、policy 校验和 bug route 识别。
3
BugAnalysisRunner
调用 feishu-bug-fetcher / meegle 获取 bug 字段和附件，再执行 startup / stuck / crash / signal / perception 分析脚本。
4
HtmlReportPublisher
发布 HTML 到 /reports/job_id/，返回 LAN 可访问链接。
5
BridgeApp → 飞书
reply 文字摘要和报告链接，群里需要时附带 HTML 文件。

H2. 时序图：回复文件消息触发直传分析

1
用户 → 飞书
先上传日志文件，再回复这条文件消息并 @bot 说“分析启动和卡顿”。
2
BridgeApp
在当前消息文本里看不到 file_key 时，沿着 reply_to / parent_id / root_id 抓被回复消息。
3
LarkClient
fetch_message(源文件消息) 取回 message payload，并抽取 file_key / image_key。
4
LogDownloader
下载时使用源文件消息的 message_id，而不是当前追问消息的 message_id。
5
BugAnalysisRunner
把这类请求按 direct_analysis 处理，后续流程与普通日志分析一致。

H3. 时序图：超级用户跨群 @bot

1
超级用户
在任意群里 @bot，不受 allowed_chats 限制。
2
Policy
命中 allowed_user_bypass，放行完整能力。
3
IntentAnalysisRunner
如果已启用，就先由本地 agent 判断是聊天、分析还是续聊。
4
执行层
根据 route 进入 OMLX、bug runner、signal、perception 或 skill runner。

H4. 时序图：reply 续聊 / 复分析

1
用户
reply 某条历史分析消息。群聊必须同时 @bot；私聊不要求 @。
2
ConversationContextStore
通过 reply_to / root_id / parent_id 找到根上下文，不再走同群最近一次兜底。
3
AgentActivityStore
读取上次会话的 job_id、report_url、progress 和 bug agent session_id。
4
执行层
bug 续聊走 run_bug_agent_followup；时间修正类走 run_bug_reanalysis；普通上下文追问走 OMLX reply_with_context。
5
BridgeApp
写回 append_exchange，延长同一 root_message_id 的上下文历史。

I. 当前边界与限制

intent_analysis 不是默认必开
不开启时，直接按 parser 的确定性规则路由。
不是所有能力都由 Agent 全权执行
bug / signal / perception 的重活仍由本地脚本负责。
外部群普通成员仍不开放通用聊天
当前只允许日志分析类能力进入。
report_server 有一个本地测试环境问题
tests/test_report_server.py 在当前环境下端口绑定会报 PermissionError。
